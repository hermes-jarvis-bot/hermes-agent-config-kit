Директива пользователя (2026-06-07):
> «если задача сложна, мы используем перепроверку независимую агентами с контекстом. наша задача
> не экономить токены, а сделать качественно, по всем правилам, работу.»

## Правило
- **Цель = качество по всем правилам, НЕ экономия токенов.** Не срезаем углы ради «дешевле/быстрее».
  Токен-бюджет — не ограничение, когда на кону корректность/стабильность.
- **Сложная / важная / необратимая задача → независимая перепроверка агентом** в **свежем
  контексте** (Generator-Evaluator): я (генератор) предлагаю решение, отдельный агент-эвалуатор
  независимо проверяет, **не доверяя моему reasoning**, имея нужный контекст (файлы, данные,
  чек-лист). Вердикт PROCEED / HOLD / REJECT.
- Чем выше моя «уверенность» без external proof — тем нужнее независимый верификатор (paradox of
  confidence).
- Для широких задач (аудит/миграция/ресёрч/review) — **fan-out воркфлоу** (десятки агентов
  параллельно), это норма, а не расточительство.

## Когда обязательно
Destructive/необратимое, архитектурные решения, merge-конфликты, security, интеграции (контракт
проверяем у получателя), «я уверен но без доказательства». Read-only/тривиальное — можно соло.

## Механизм
`Agent` tool со свежим контекстом / `Workflow` для fan-out. Шаблон вердикта и источники силы —
`no-guessing.md` (Independent Verifier Agent). База — CLAUDE.md «Generator-Evaluator Pattern»,
skill `harness-design`, skill `workflow-orchestration`.
